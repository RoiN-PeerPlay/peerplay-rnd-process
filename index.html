Story Breakdown

How We Break Down Features

📖 Story / Task Business or Technical Topic One complete topic — the top-level unit of work
🔧 Dev Sub-Task
🧪 QA Sub-Task
🐛 Bug / Story Bug
🎨 Art / 3rd Party
Rule 1: Each Story = one business or technical topic. It is the top-level unit — not a subtask of anything else.
Rule 2: Time estimates live on the Dev & QA subtasks, not on the Story itself.

Action Plan

Steps to Get There

1
Setup Configure Jira Bug Workflow
New bug statuses: To Do → In Progress → In Review → Ready for QA → Testing → Passed QA → Ready for Release → Close
2
Setup Define Story / Task Structure in Jira
Story/Task as top-level unit • Dev Subtask + QA Subtask • Estimates on subtasks only
3
Integrate TLD Template in Notion
Technical Design Doc template ready for Dev to use from day one
4
Integrate Testomatio + Notion AI Integration
AI auto-generates test cases from PRD & TLD — QA starts with a ready-made test plan
5
Integrate GitHub ↔ Jira PR Linking
Every PR references a Jira story or bug — no orphan code changes
6
Execute Team Alignment Session
Walk Product, Dev & QA through the full new process together
7
Execute First Feature Under New Process
Run end-to-end: PRD → TLD → TDR → Stories → Dev+QA → Release
8
Improve Retrospective & Lock It In
What worked, what didn't — refine and make the process permanent
Same team. Clearer process. Better games. 🎮
